# Parvathy PhD: Vietnam Stroke Facility Replication

This folder is the fresh-data replication workspace for the Vietnam stroke-facility allocation case. It deliberately does **not** use Fleur's historical `.npy` or pickle data. The older Fleur notebooks remain useful for algorithmic inspiration, but all executable experiments here are based on data produced by the current PISA distance pipeline.

Repository code and documentation live here. Raw downloads, pipeline outputs, logs, optimization instances, and experiment traces are written outside the repository under:

```text
C:\local\Parvathy\Vietnam
```

The PISA pipeline itself is imported unchanged from:

```text
Research-Sandbox/general_distances_per_country
```

The heuristic implementation follows the scalable spirit of:

```text
Research-Sandbox/approximated_tradeoff
```

It uses compressed sparse row style biadjacency arrays, in-place NumPy coverage updates, first-improvement local search, and budgeted constructive heuristics. GRASP and path relinking are added on top of the same data structures rather than on notebook-era row/column pickle dictionaries.

The Vietnam runners default to the local speed-safe variants `--local-search first_sparse` and `--path-relinking-method fast`. The original `approximated_tradeoff` first-swap local search and original path relinking remain available for validation runs.

## Data Rule

Use only current PISA outputs for replication runs:

- WorldPop / Meta population points generated by the PISA pipeline;
- Geofabrik OSM road network processed by the PISA pipeline;
- the current Vietnam stroke table as a `table` source layer;
- candidate sites generated by the PISA candidate-grid layer;
- distance matrices written by the PISA pipeline.

Do not run experiments from historical Fleur `.npy`, `.npz`, `.pkl`, or notebook-local data files.

## Step 1: Fresh Candidate-Grid PISA Run

The candidate-grid run should include both the current stroke table and generated candidates as source layers. Use split sparse matrices so the max-cover instance builder can load only the table-to-population and candidate-to-population pieces it needs.

```powershell
cd C:\local\GIT\Public-Infrastructure-Service-Access\Research-Sandbox\Parvathy_PhD\Vietnam
py scripts\run_vietnam_fresh.py vietnam `
  --sources table candidates `

  --source-lon-column longitude `

  --log-file C:\local\Parvathy\Vietnam\logs\fresh_table_candidate_10kmgrid_150km.log
```

For a short pipeline smoke test, add a bounding box and/or a small `--max-points` value. The full national candidate-grid run is the one to use for reported results.

## Step 2: Build A Fresh Max-Cover Instance

The builder converts PISA parquets into a CSR `.npz` instance compatible with `approximated_tradeoff` data structures. Table sources are treated as existing stroke facilities. Demand already covered by existing facilities at the threshold is assigned zero incremental weight; reported results add the existing baseline coverage back afterward.

```powershell
py scripts\build_pisa_instance.py `
  --outputs-dir C:\local\Parvathy\Vietnam\fresh_downloads\vietnam_data\outputs `
  --run-tag-marker candidates_spacing_10000_maxsnap_5000 `
  --threshold-m 100000 `
  --output-npz C:\local\Parvathy\Vietnam\optimization\vietnam_10kmgrid_100km_threshold.npz
```

Build separate instances for the thresholds needed to approximate Fleur's experiment design, for example 20 km, 50 km, and 100 km, provided the pipeline matrix was retained with a large enough `--max-total-dist`.
